01
What Is a Network, Actually?
Foundation
The Big Picture

A network is any group of devices that can communicate. Your laptop, a printer down the hall, Harvard's servers — all nodes. The internet is just a massive network of networks tied together by agreed-upon rules called protocols.

🏙️ City Analogy

Buildings = devices. Streets = cables and Wi-Fi. Addresses = IP addresses. The post office = DNS. Traffic rules = protocols. Simple as that.

The 6 Concepts You Must Know
📍
IP Address
Logical address used to route data across a network. Assigned dynamically — can change.
10.0.2.55
🪪
MAC Address
Physical ID burned into the network card at the factory. Permanent. Unique to that hardware.
A4:83:E7:2C:11:F0
📖
DNS
Translates domain names to IP addresses. You type google.com, DNS finds the number.
8.8.8.8
🏷️
DHCP
Server that hands out IP addresses automatically when a device connects. Nothing to configure.
Auto-assign
🚪
Gateway
The router connecting your local network to the rest of the internet. All outbound traffic flows through it.
10.0.0.1 (typical)
🔒
Subnet
A segmented slice of a larger network. Harvard separates departments into their own subnets.
/24 = 254 hosts

MAC addresses only work on your local segment. IP addresses work everywhere. ClearPass identifies your hardware by MAC — that's why every device needs its own registration.

✓ Key Takeaway

IP = logical, can change. MAC = physical, permanent. ClearPass authenticates devices by MAC address, not IP.

03
Finding Your MAC Address
Hands-On

You need the device's Wi-Fi MAC address before you can register it. Here's how to find it on everything you'll touch at Harvard.

Method A — System Settings
  1. Click the Apple menu → System Settings
  2. Click Wi-Fi in the sidebar
  3. Click Details… next to your connected network
  4. Look for Wi-Fi Address — format: A4:83:E7:2C:11:F0
Method B — Terminal (Faster)
  1. Spotlight → type terminal → open it
  2. Run: ifconfig en0 | grep ether
  3. The ether line is your Wi-Fi MAC address
💡 Pro Tip

en0 is almost always Wi-Fi on Macs. Ethernet adapters are usually en1 or higher. When in doubt, confirm in System Settings first.

Method A — Settings
  1. Press Win + I to open Settings
  2. Go to Network & Internet → Wi-Fi
  3. Click your connected network name
  4. Scroll to find Physical address (MAC)
Method B — Command Prompt (Faster)
  1. Press Win + R → type cmd → Enter
  2. Run: ipconfig /all
  3. Find Wireless LAN adapter Wi-Fi section
  4. Look for Physical Address — that's your MAC
💡 Dell Shortcut

Many Dell laptops have a sticker on the bottom labeled "WLAN MAC" — useful when the device won't boot or is locked out of the network entirely.

  1. Go to Settings → General → About
  2. Scroll to find Wi-Fi Address
  3. That's your MAC — copy it before going to the portal
⚠ Critical — Private Wi-Fi Address

iOS rotates the MAC address per network by default. This BREAKS ClearPass. Before registering an iPhone: Settings → Wi-Fi → ⓘ next to Harvard Secure → toggle Private Wi-Fi Address OFF. Without this step, the device will keep losing access.

APIPA — When Your IP Starts with 169.254
Diagnostics
What Is APIPA?

APIPA stands for Automatic Private IP Addressing. When a device can't reach a DHCP server to get a real IP address, Windows automatically assigns itself an address in the range 169.254.0.1 to 169.254.255.254. This is a self-assigned fallback — it means the device has no valid network connection.

⚠ What It Looks Like

If you run ipconfig and see an address starting with 169.254 — that's APIPA. The device cannot reach the internet or Harvard resources. This is always a symptom, never a solution.

Normal IP
10.243.16.42
Assigned by DHCP server ✓
→
APIPA (Problem!)
169.254.44.17
Self-assigned — no DHCP ✗
Why Does It Happen?
🔌
Not plugged in
Ethernet cable disconnected or not seated properly. Check both ends.
🔒
Port not registered
Ethernet port not registered in ClearPass — device isn't authorized on the network.
📡
DHCP server unreachable
Network infrastructure issue — DHCP server down or the device is on an isolated VLAN.
🔁
Driver/adapter issue
NIC driver problem or adapter malfunction preventing proper DHCP negotiation.
How to Fix It
  1. Run ipconfig /release then ipconfig /renew — forces the device to re-request an IP from DHCP
  2. Check the physical connection — reseat the Ethernet cable if wired, or reconnect to Wi-Fi
  3. If wired, verify the port is registered in ClearPass or tools.noc.harvard.edu
  4. Restart the network adapter: netsh interface set interface "Wi-Fi" disable then enable
  5. If none of the above work, escalate to the HUIT networking team with the MAC address and port label
✓ Quick Memory Aid

169.254 = "I gave up waiting for DHCP." Always signals a connectivity failure, not a valid network state.

Ping, Latency & Packet Loss
Diagnostics
What Does Ping Do?

Ping sends a small packet called an ICMP Echo Request to a target address and waits for an ICMP Echo Reply. It measures how long that round trip takes — this is your Round Trip Time (RTT), measured in milliseconds (ms). It's the most fundamental network diagnostic tool you have.

→Request: Your device → 8.8.8.8
→Reply: 8.8.8.8 → Your device: time=12ms TTL=118
→Request: Your device → 8.8.8.8
→Reply: 8.8.8.8 → Your device: time=11ms TTL=118
→Request: Your device → 8.8.8.8
→Request timed out — Packet Lost!

On campus hitting 8.8.8.8, you should see under 20ms. If you're seeing 100ms+ to Google on a wired connection, something is wrong with the path — could be congested uplink, routing issue, or a faulty cable.

Packet Loss

Packet loss is when packets you send don't come back. Even 1–2% packet loss causes significant issues for video calls, VoIP, and real-time applications. Look at the summary line ping prints when it stops:

Packets: Sent = 100, Received = 97, Lost = 3 (3% loss)

RSSI — How Strong Is Your Wi-Fi Signal?
Wi-Fi
What Is RSSI?

RSSI (Received Signal Strength Indicator) measures how strong the Wi-Fi signal is between a device and the access point. It's measured in dBm (decibel-milliwatts) — a negative number where closer to zero = stronger. -30 dBm is excellent; -90 dBm is practically unusable.

Checking RSSI on Windows

Windows reports signal as a percentage in most UIs, but you can get the real dBm value:

  1. Open PowerShell or CMD and run: netsh wlan show interfaces
  2. Look for the Signal field — this is a 0–100% value (not dBm directly)
  3. For raw dBm, run: (netsh wlan show interfaces) -match "Signal"
  4. For detailed RSSI in dBm, use: Get-NetAdapter | Get-NetAdapterStatistics or check Wireless Diagnostics tools

Windows % is a rough estimate: 100% ≈ -50 dBm, 75% ≈ -67 dBm, 50% ≈ -75 dBm, 25% ≈ -85 dBm. Below 50% (under ~-75 dBm), users will have a bad experience.

Wall Jacks & Ethernet Ports
Infrastructure
Reading a Wall Jack Label

Every Ethernet wall jack in Harvard buildings has a label that tells you exactly which switch port it connects to. You'll need to read this when a user has a wired connectivity issue — the networking team needs this information to locate and troubleshoot the specific port.

D1-04-A
D1
Jack Type — D1 = Data port 1, D2 = Data port 2 on the same plate. D1 is typically the primary data jack. Voice (V) or Aux (A) may appear on other ports.
04
Port Number — The specific port on the patch panel in the telecommunications closet (IDF/MDF). This maps to a physical switch port.
A
Panel / Zone — Identifies which patch panel or zone in the closet. Letters increment as more panels are added: A, B, C...
📋 What to Capture for a Ticket

Read the full label off the wall plate — something like D1-04-A or D2-12-B. Write it down exactly as it appears. Include the room number. This is what the networking team needs to locate the exact switch port and check its status.

D1 vs D2 — Which to Use
🔵
D1 — Primary Data
The main data jack on the plate. Always try D1 first. Should be registered in ClearPass and active for standard workstation use.
🟡
D2 — Secondary Data
Second data jack on the same plate. May be on a different switch port or VLAN. Useful as a fallback or for connecting a second device.

Opening a HUIT Networking Ticket
Process
Collect This Before You Open Any Ticket

Before reaching out to the HUIT networking team, you need to gather specific information. Incomplete tickets get bounced back, which delays resolution for the user. Spend 60 seconds collecting this first.

MAC Address of the affected device
Run ipconfig /all (Windows) or ifconfig en0 | grep ether (Mac). Copy the Physical Address — Wi-Fi MAC for wireless issues, Ethernet MAC for wired issues.
Issue description — what is the user experiencing?
Be specific: "No connectivity", "169.254 APIPA address", "Connected but can't reach Harvard resources", "Intermittent drops every 20 minutes", "Slow speed — 2 Mbps on speedtest". Vague tickets like "can't connect" get deprioritized.
Wired or wireless?
This determines the entire troubleshooting path. For wired: include the wall jack label. For wireless: include the AP name and RSSI if available.
Nearest AP name (wireless issues)
Read the label on the closest Aruba AP — format is like paine-library-ap1. The networking team can pull up that AP in their management console and see its status, client list, and error logs immediately.
Wall jack label (wired issues)
The full label from the wall plate — e.g., D1-04-A. Include the room number and building. The team uses this to locate the exact switch port.
Building, floor, and room number
Physical location. This helps the networking team dispatch someone if a site visit is needed and identifies which network equipment serves that area.
When did it start? Was anything recently changed?
New device? Moved desks? Just ran Windows update? These details narrow down the cause dramatically. "It stopped working Monday morning" is far more useful than "it hasn't worked in a while."
What have you already tried?
ipconfig /release /renew? Forgot and re-joined the network? Tested with a different cable? List what you've done so the networking team doesn't repeat your steps.
